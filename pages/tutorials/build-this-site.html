用AI从零搭建一个AI编程攻略网站

本站诞生记 — 一个真实的AI辅助开发全记录

📅 2026-05-14 ⏱ 20分钟 中级
Claude AI编程 实战 案例

项目缘起

说来也有趣，这个网站本身就是一个"用AI做AI教程"的元实践。过去一年多，我目睹了AI编程工具以惊人的速度演进——从GitHub Copilot补全代码片段，到Claude和GPT能一口气生成整个组件，再到Cursor这样的AI-native编辑器彻底改变开发体验。几乎每周都有新工具、新技巧涌现。

但我发现一个问题：中文世界里缺少一个系统化的AI编程攻略站。信息散落在知乎专栏、公众号文章、B站视频和各个技术博客里，不成体系。新手开发者想入门AI编程，往往不知道从哪里开始——是先学Prompt Engineering？还是直接上手Cursor？市面上那么多AI工具，哪个最适合自己的场景？

于是我想，为什么不自己做一个呢？一个聚焦"AI编程"的中文内容站，覆盖从入门到进阶的完整学习路径。更重要的是——这个网站本身就应该用AI来建，用实践来证明AI编程的真实能力。这就是"AI编程指南"（ai-coding-guide）的诞生起源。

我跟Claude说："我想做一个AI编程攻略的中文网站，帮我设计一下。"它没有敷衍地只给建议，而是直接给出了完整的网站架构、技术方案和内容规划。那个瞬间我就意识到——这件事确实能成。

需求分析与规划

动手之前，我先花时间梳理了目标用户群体和他们的核心需求。这不是一个泛泛的技术博客，而是一个有明确使命的内容站。我让AI帮我做了详细的需求分析：

目标用户画像

我梳理出三类核心读者：

  • 编程新手（占比约40%）：没有太多开发经验，想知道AI能否帮他们跨过编程门槛，能否"不会代码也能做东西"
  • 在职开发者（占比约35%）：已经会用传统方式编程，想了解AI工具如何提升工作效率，如何融入日常工作流
  • 技术决策者（占比约25%）：技术负责人或创业者，关注AI编程的工具选型、团队落地和最佳实践

这三类人的需求差异很大。新手要"手把手教程"，开发者要"高效技巧"和"工具对比"，决策者要"案例"和"方法论"。一个单一维度的内容结构无法同时满足所有人，所以我在内容策略上做了分层设计。

内容策略

我决定用"两条腿走路"的策略：

  1. 纵向分层：从入门到进阶，每篇文章标注难度等级，让读者能找到适合自己的起点
  2. 横向分类：按主题划分栏目，每个栏目聚焦一个维度，方便定向查找

第一阶段的目标是覆盖所有核心栏目，每个栏目至少2-3篇文章，形成基础内容矩阵。有了骨架之后，再根据读者反馈和数据表现，逐步填充和优化。

信息架构设计

确定了目标用户和内容策略后，接下来就是设计网站的信息架构。这是搭建过程中的关键环节——结构决定了用户能否高效地找到所需内容。

我采用了6大栏目体系：

栏目定位面向人群
新手入门零基础起步，第一篇文章教你30分钟做出第一个网页编程新手
工具评测深度对比主流AI编程工具（Cursor、Claude Code、Copilot等）所有开发者
实战教程完整项目实战，从需求到上线的全流程演示有基础想进阶的开发者
技巧提升Prompt Engineering、效率技巧、工作流优化在职开发者
经验分享真实案例复盘，踩坑记录，团队落地经验技术决策者、进阶开发者
资源工具汇总、推荐阅读、关于本站所有人

这个六栏体系的设计逻辑是：覆盖用户从"听说AI编程"到"精通AI编程"的完整旅程。"新手入门"降低了门槛，让任何人都能快速获得正反馈；"工具评测"帮助用户做出技术选型；"实战教程"展示真实开发场景；"技巧提升"和"经验分享"则通向高阶应用。

在文章粒度的规划上，我设定了首批14篇文章作为站点基础内容。每篇文章控制在10-25分钟的阅读时长，确保信息密度适中。每篇文章还配置了标签系统（如"Claude""Cursor""Prompt"等），便于跨栏目关联和搜索。

技术选型

技术选型上，我的原则是"足够简单，但不简陋"。作为一个内容站，不需要复杂的后端架构，但必须在用户体验上做到位。

为什么选择纯静态HTML？

很多朋友问我："为啥不用Next.js或Hugo？静态页面一个一个写不累吗？"答案很简单——因为我有AI。AI生成静态HTML的效率极高，我只需要描述需求，它就能在几秒内输出一个完整的页面。既然AI代劳了重复劳动，"手动写HTML费时"就不再是问题了。

而且纯静态方案带来的优势很明显：

  • 零构建步骤：写完直接扔服务器，不需要编译、打包
  • 极速加载：没有JS框架的运行时开销，首屏加载几乎瞬间完成
  • 部署灵活：GitHub Pages、Vercel、Netlify，任意静态托管服务都能托管
  • 搜索引擎友好：天然SEO，内容直接暴露在HTML中

技术栈全景

技术用途选择理由
Tailwind CSS (CDN)样式框架零构建、响应式、暗色模式原生支持
highlight.js代码高亮轻量级、支持语言多、CDN可引用
Fuse.js (搜索)客户端全文搜索无需后端服务，模糊匹配体验好
Noto Sans SC中文字体Google出品，美观且免费
Vercel托管全球CDN、自动HTTPS、Git集成

特别值得一提的是Tailwind CSS 的 CDN 方案。很多人不知道Tailwind可以直接通过CDN引用，不需要npm安装和构建配置。虽然CDN模式不支持自定义配置的Tree Shaking，但对于一个内容站来说完全够用。配合darkMode: 'class'配置，暗色模式的实现也非常优雅。

AI辅助开发工作流

终于到了本文最核心的部分——我是如何用AI来搭建这个网站的。整个流程可以概括为4个阶段。

阶段一：AI分析需求，设计方案

第一步，我向Claude描述了想做的东西：一个中文AI编程攻略网站，目标用户是谁，希望包含哪些内容。Claude没有直接开始写代码，而是先帮我梳理了需求、设计了信息架构，还画出了页面布局草案。

这个环节的关键在于：不要把AI当成打字员，而是当成架构师。先让它理解全局，再让它生成代码，这样产出的东西才有整体感。

阶段二：AI生成模板和页面骨架

确定了架构之后，我开始让Claude生成具体的HTML文件。我使用了一个"模板+填充"的策略：先让AI生成一个标准文章模板，包含导航栏、文章布局、侧边栏TOC、搜索弹窗、页脚等完整结构。然后每写一篇新文章时，基于这个模板来生成。

这是我的核心工作流：

  1. 描述文章主题：告诉AI要写什么类型的文章、面向谁、核心要点是什么
  2. AI生成初稿：Claude输出完整的HTML文件，包括内容和样式
  3. 我审阅和修改：通读内容，修正事实性错误，调整表达方式
  4. AI优化：根据反馈进行润色和改进
  5. 整合上线：加入导航索引、搜索索引，部署到Vercel验证

多篇文章我甚至让多个AI"并行工作"——Claude处理架构和技术内容，另一路生成工具评测类文章，我在中间做协调和质量把控。这种"多Agent协作"的模式大幅加速了内容生产。

阶段三：搜索和交互功能

站点需要一个搜索功能。我没有引入Elasticsearch或Algolia这样重型的方案，而是用Fuse.js做了一个纯客户端模糊搜索。AI帮我生成了搜索索引JSON文件（包含所有文章的标题、描述、路径和标签），以及搜索弹窗的前端交互逻辑。整个过程只需要几十行代码。

阶段四：调试和打磨

没有任何项目是一次就完美的。在搭建过程中，我遇到了一些问题，比如：

  • 暗色模式切换时图标没有正确切换 → 让AI检查并修复了JS逻辑
  • 移动端导航菜单的展开/收起动画不平滑 → AI调整了CSS过渡
  • 搜索结果的展示格式不够直观 → 优化了渲染模板
  • 代码块在文章中的样式与网站主题不匹配 → 调整了highlight.js主题和自定义CSS

每一个问题，我都是直接描述现象，AI给出修复方案。这种"描述问题→获得修复→验证"的循环非常高效。

核心Prompt分享

很多读者问我是怎么向AI描述需求的。这里分享几个实际使用过的Prompt，供大家参考。

需求分析Prompt

这是最初让AI帮我做网站规划时使用的Prompt：

你是一个资深的网站架构师和产品经理。我想搭建一个中文的"AI编程攻略"网站，
目标用户是中文开发者（从新手到高级），核心主题是"用AI辅助编程"。

请帮我完成以下工作：
1. 分析目标用户的分类和各自的核心需求
2. 设计网站的信息架构（栏目体系）
3. 规划首批内容选题（10-15篇）
4. 推荐技术栈（要求：纯前端、零后端、可部署在Vercel）

输出格式：请用表格和列表清晰呈现，不要用大段文字。

这个Prompt的关键在于：定义了角色（网站架构师）、目标（AI编程攻略网站）、约束条件（纯前端、Vercel），并指定了输出格式。AI在明确约束下给出的方案，远比开放式的提问要具体和可用。

文章生成Prompt模板

这是我生成每篇文章时使用的模板化Prompt：

请生成一篇关于[主题]的HTML文章，用于我的AI编程攻略网站。

文章信息：
- 标题：[文章标题]
- 描述：[一句话描述]
- 难度：[初级/中级/高级]
- 阅读时间：[X分钟]
- 标签：[标签1, 标签2, ...]

技术规范：
- 使用Tailwind CSS（CDN版本）
- 代码块使用highlight.js（语言请标注正确）
- 文章内容放在 <div class="article-content max-w-none"> 内
- 中文内容，专业但易懂
- 包含H2/H3多层标题结构
- 文章长度：800-1200字

写作要求：
- 第一段要吸引人，可以用一个具体场景或问题开头
- 每个要点都配具体例子
- 代码示例要完整可运行
- 结尾要有总结段落

这个模板的核心设计思路是：把"写作规范"和"技术规范"分离。AI在处理多重要求时容易顾此失彼，分开描述能让它在每个维度上都做到位。

调试和修复Prompt

当遇到问题时，我通常会这样描述：

有一个Bug需要修复：
- 页面：[页面路径]
- 问题描述：在Chrome浏览器中，点击暗色模式切换按钮后，
  太阳/月亮图标没有正确切换。当前图标始终显示为月亮。
- 相关代码：js/main.js 中的 dark-toggle 事件处理逻辑
- 期望行为：点击后图标从月亮切换为太阳（亮色模式），再点击切换回来

请先分析可能的原因，然后给出修复代码。

注意这个Prompt的结构：问题现象 + 相关代码位置 + 期望行为 + 分析要求。让AI先分析再给代码，比直接让它"修复"效果更好。

成果与数据

经过几天的集中搭建，网站的第一阶段顺利上线。以下是一些关键数据：

  • 29个HTML页面，涵盖首页、6个栏目页、14+篇文章以及辅助页面
  • 全套交互功能：暗色模式切换、响应式导航、全文搜索、代码高亮、阅读进度条、回到顶部
  • 所有页面HTTP 200：经过完整验证，无死链和404
  • 平均加载时间<1s：纯静态页面+CDN分发，用户体验流畅

更重要的是，整个开发过程中，我没有手动写过一行CSS——所有的样式都由AI通过Tailwind类名生成。我没有配置过任何构建工具——CDN方案省去了Webpack/Vite的配置步骤。我没有写过搜索后端——Fuse.js在前端就完成了所有工作。

这让我能够把精力完全集中在"内容"和"体验"上——思考哪些文章对读者最有价值，如何让导航更直观，怎样让代码示例更容易理解。这才是内容创作者应该关注的事情。

经验与反思

做完这个项目，我有很多感触想分享。以下是几个最重要的发现：

什么做得好

AI在内容生成上的表现超出预期。最初我担心的"AI写的内容会不会很空洞"其实多虑了——只要Prompt给得具体（包括目标读者、核心观点、参考风格），AI产出的内容骨架是非常扎实的。我只需要做"编辑"而非"创作"，效率提升了数倍。

多Agent并行工作流有效。让不同的AI会话负责不同的任务——一个处理架构，一个生成文章，一个做代码检查——彼此独立，我作为"总编辑"统一把控质量。这种方式比一个人从头做到尾高效得多。

"模板化+个性化"的内容策略可行。统一的页面模板保证了体验的一致性，而每篇文章的个性化内容保证了阅读价值。AI特别擅长在固定框架内做创造性填充。

顺便提一句，你现在读的这篇文章本身就是个套娃——用AI写了一篇关于如何用AI建站的文章，发布在用AI搭建的网站上。如果觉得有点晕，别担心，写的人也有点晕。

遇到的挑战

内容质量控制是最难的部分。AI生成的内容有时会出现事实性错误（比如某个工具的版本号不对，或者某个功能描述有偏差）。每篇文章我都需要通读一遍，特别是技术细节部分必须逐个核实。这个步骤不能省。

多个AI会话之间的风格一致性是个问题。不同的会话生成的页面在措辞风格、表达习惯上有微妙差异。我花了一些时间来统一，方法是在每个新的AI会话中都粘贴"风格指南"——包括之前文章的段落样例和写作规范。

Prompt的设计需要持续迭代。一开始我写的Prompt比较笼统，输出质量不稳定。后来我学会了把复杂需求拆解成多个简单Prompt，每次都只专注一个目标。这个"从粗放到精准"的过程本身就是Prompt Engineering的实战练习。

给后来者的建议

如果你也想用AI搭建一个网站，我的建议是：

  1. 先想清楚"为什么"再动手。AI能帮你解决"怎么做"，但"为什么做"需要你自己想清楚。清晰的目标是有效Prompt的基础。
  2. 把AI当成协作者，而不是工具。当你对AI说"你觉得这个结构怎么样"而不是"按这个结构生成"，你会得到更有价值的建议。
  3. 保留人工审核环节。AI输出的内容一定要经过人工审阅，特别是涉及数据、日期、版本号等事实性信息。
  4. 从小处开始，快速迭代。不需要一次性规划所有内容，先发布一个最小可用版本，然后根据反馈持续优化。

💡 本文总结

这个网站本身就是AI编程能力的最好证明。从需求分析到架构设计，从页面生成到内容创作，AI贯穿了整个开发流程。希望这个案例能启发你，用AI去实现自己的想法。
