非程序员用AI做出上线产品的真实经历

📅 2026-05-14 ⏱ 10分钟 入门
案例非程序员

背景：一个产品经理的"疯狂"想法

我是某互联网公司的产品经理，日常工作就是画原型、写PRD、跟研发撕需求。做了五年产品经理，我对代码的了解仅限于HTML标签和几句简单的SQL查询。长久以来我有一个想法——做一个面向小型团队的OKR管理工具。

市面上现有的OKR工具要么功能太重（适合千人以上的大公司），要么价格太贵（SaaS按人头收费让十人以下的团队望而却步）。我的目标是做一个轻量级、免费、开箱即用的OKR管理工具，专门服务5-50人的小型团队。

放在以前，这个想法只能停留在原型图上，然后排队等公司的研发资源——而这通常意味着"永远不会被排上"。但在2025年底，AI编程工具的爆发让我看到了一种全新的可能性：为什么不自己试试？

选型过程：哪个AI工具最适合我

我花了两周时间调研和试用各种AI编程工具，最终的选择如下：

  • Cursor： 我最终选择的主力编辑器。它的Composer功能可以同时修改多个文件，Tab补全速度很快。最关键的是，它对非程序员比较友好——我只需要用自然语言描述需求，它就能生成代码。
  • Claude (通过API)： 用于处理复杂的逻辑问题和代码审查。当我遇到Cursor生成的代码有问题时，会把代码粘贴给Claude让它分析。
  • GitHub Copilot： 辅助补全，但后来发现Cursor的体验更好，就放弃了。
  • Vercel： 用于部署。它和GitHub的集成非常方便，push代码后自动部署。

为什么选择Cursor而不是其他工具？因为Cursor的"Agent模式"能理解整个项目的上下文，而不仅仅是当前文件。对于我这样的新手来说，不需要了解项目的完整架构，只需要告诉它"我要加一个功能"，它就会自动创建必要的文件和修改相关的代码。

项目实战：从零开始的旅程

第一阶段：搭起骨架

说实话，我连Next.js是什么都不知道。我的第一个Prompt非常简单：

帮我创建一个Next.js项目，是一个OKR管理工具。需要有：
1. 用户登录和注册功能
2. 创建OKR的面板
3. 展示OKR的看板视图
使用Tailwind CSS做样式，数据库用PostgreSQL。

Cursor帮我创建了完整的项目结构，包括pages、components、API routes等。我当时完全看不懂这些文件是干什么的，但按照Cursor的指示运行了 npm run dev 后，浏览器里真的出现了一个可以交互的页面！

# Cursor告诉我执行的命令
npx create-next-app okr-tool --typescript --tailwind
cd okr-tool
npm run dev

第二阶段：核心功能迭代

项目骨架搭好后，我开始逐个功能迭代。这个过程基本上是：想一个功能 -> 写Prompt描述功能 -> 让Cursor生成代码 -> 运行看效果 -> 发现问题 -> 继续对话修改。

举个例子，实现OKR的"关键结果"进度更新功能时，我的Prompt是这样的：

在OKR详情页面，每个Key Result后面添加一个进度更新按钮。点击按钮弹出对话框，可以输入：
1. 当前进度（百分比，滑块选择）
2. 更新备注（文本）
3. 信心指数（高中低三档）

保存后，进度数据要存入数据库，并在OKR列表页面显示最新的进度和信心指数。
对话框用Modal组件实现。

Cursor几分钟就生成了完整的代码，包括前端组件、API路由和数据库迁移文件。虽然第一次生成的Modal样式有点丑，我接着说了句"把Modal做得好看一点，加个动画效果"，它就帮我改好了。

第三阶段：上线部署

部署对我来说是最神秘的部分。Cursor指导我完成了以下步骤：

  1. 在GitHub创建仓库，将代码推送到远程
  2. 在Vercel上导入GitHub仓库
  3. 在Vercel上配置环境变量（数据库连接字符串等）
  4. 使用Neon（一个免费的PostgreSQL云服务）创建数据库

整个过程大概花了两小时，其中大部分时间是在等域名解析生效。当看到自己的产品在浏览器中通过真实域名访问的那一刻，说实话我激动得差点从椅子上跳起来。

遇到的困难

整个过程并非一帆风顺，我遇到了几个主要的挑战：

1. 调试对我来说是黑箱。 当页面报错时，作为一个非程序员，我完全看不懂错误信息。我的策略是把整个错误信息复制给AI，问它"这是什么意思，该怎么修"。AI通常会给出修复步骤，我照着做就好。偶尔AI给出的方案也不对，就需要换一种问法重新问。

2. 数据库设计需要业务判断。 AI在设计数据库表时有时不太合理。例如它会把所有字段都放在一张表里，忽略了规范化。这时候我需要用自己的产品经验来判断数据模型是否合理，然后告诉AI调整。

3. 状态管理越来越复杂。 随着功能增多，不同页面间的数据共享变得复杂。我完全不知道"状态管理"这个概念，是AI告诉我需要使用React Context或Zustand。我让AI帮我实现了Zustand store，从那之后状态问题少了很多。

4. 版本控制的概念门槛。 Git对我来说是一个全新的概念。我学会了几个基本命令（add、commit、push、pull），但遇到合并冲突时还是需要AI手把手教我怎么处理。

最终成果

历时三个月（业余时间），我成功上线了第一版OKR工具。以下是最终产品的核心功能：

  • 用户注册/登录（邮箱+密码，支持Google OAuth）
  • 团队管理（创建团队、邀请成员）
  • OKR创建与编辑（支持季度和年度周期）
  • 关键结果进度追踪（百分比+备注+信心指数）
  • 看板视图（拖拽排序）
  • 历史记录与回顾（每个OKR的变更历史）
  • 数据导出（Excel格式）

目前已有3个真实团队在使用这个工具，累计约20个用户。虽然用户量不大，但每一个用户都是真实注册、真实使用的——对于一个非程序员用AI做出来的产品来说，这已经是最有力的肯定了。

给其他非技术背景者的建议

如果你也是非程序员，正在考虑用AI编程来实现自己的产品想法，以下是我最真诚的建议：

1. 从一个非常小的项目开始。 不要一上来就想做"中国的Shopify"或"下一代Notion"。选择一个足够小的、用户需求真实且明确的功能点，先做出来，再逐步迭代。

2. 产品的核心是需求理解，不是技术实现。 作为非程序员，你对用户需求的理解和产品设计的能力就是最大的优势。AI负责"怎么做"，你负责"做什么"——这个分工很清晰。

3. 学会阅读错误信息。 你不需要理解每一行代码的含义，但至少要学会复制错误信息给AI，并能够执行AI给出的修复指令。

4. 小步快跑，频繁测试。 每完成一个小功能就运行一下看看效果，不要攒了一大堆代码再测试。这样出了问题更容易定位。

5. 准备好被拒绝。 AI生成代码的成功率不是100%，你需要有耐心去调整Prompt描述，甚至有时候需要完全换一种思路重新开始。

6. 善用AI来学习。 每次AI解释了某个技术概念或生成了某段代码，花几分钟理解一下。慢慢地你会发现，你正在不知不觉中学会编程——这不是目标，但它是AI编程时代给非程序员的美妙副产品。

最后，我想说的是：技术从未如此平权。十年前，把产品想法变成现实需要组建一个完整的工程团队；现在，你只需要一个AI和一个敢于尝试的心。不要等到"学会编程"再开始，用AI编程最大的秘诀就是——直接开始。

💡 本文总结

AI编程工具让非技术背景的人也能实现产品想法。这个真实案例展示了从零到上线的完整旅程。
